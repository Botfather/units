First principles

What Units is and is not

✓

Units is

  • ◆A minimal DSL for interactive UI trees
  • ◆A parser + runtime with low syntax overhead
  • ◆A compact notation for humans and LLMs alike
  • ◆Compatible with React and custom renderer hosts
✗

Units is not

  • ◆A replacement for all React patterns
  • ◆A magical runtime optimizer
  • ◆A sandbox for untrusted JS expressions
  • ◆Right when teams want logic-dense inline JSX

DSL explained

A small language for one job

Think of a DSL as instructions for building with blocks — focused on one domain with extra words stripped out so structure stays clear and repeatable.

In plain terms

JSX is general JavaScript + UI syntax. Units is a smaller, UI-first notation — you trade generality for compactness when the job is "describe a UI tree."

inbox.ui
Card {
  Header { text 'Inbox' }
  Body   { text @message.preview }
}

LLM token perspective

Why fewer tokens matters

What token count affects

  • ◆How much context fits in the model window
  • ◆How expensive each generation step is
  • ◆How noisy diffs become after small edits
  • ◆How stable outputs are across retries

Why JSX can cost more

JSX repeats open/close tags, braces, callback wrappers, and attribute ceremony. Units compresses the same tree shape with fewer structural symbols.

Run pnpm bench:react-vs-dsl for measured results.

Side-by-side

JSX vs Units DSL

JSX
↑ more tokens 
function TodoList({items, onSelect}) {
  return (
    <section className="panel">
      <h2>Tasks</h2>
      <ul>
        {items.map((item) => (
          <li key={item.id}>
            <button
              onClick={() => onSelect(item.id)}
            >
              {item.title}
            </button>
          </li>
        ))}
      </ul>
    </section>
  );
}
Units DSL
↓ fewer tokens 
Section (class:'panel') {
  H2 { text 'Tasks' }
  Ul {
    #for (item in @items) {
      #key (@item.id)
      Li {
        Button
          (!click { onSelect(@item.id) }) {
          @item.title
        }
      }
    }
  }
}

Where Units fits

Choose by workflow, not hype

Strong fit

  • ◆LLM-assisted UI generation where token budget matters
  • ◆Template-heavy products with repeated structural patterns
  • ◆Pipelines needing parseable UI representations
  • ◆Teams valuing compact syntax and explicit control flow

Weak fit

  • ◆Apps with logic deeply coupled to arbitrary JSX code
  • ◆Use cases requiring sandboxed expression execution
  • ◆Teams that want zero extra language surface to learn
  • ◆Projects where token cost is not a concern

Grammar reference

The complete grammar

Expressions are raw JS slices — not parsed by the DSL parser itself.

grammar.ebnf
Document  := Node*
Node      := TagNode | TextNode | ExprNode | Directive
TagNode   := Ident Props? Children?
Props     := "(" Prop (","? Prop)* ")"
Prop      := Ident (":" Value | "=" Expr | "?=" Expr) | Event
Event     := "!" Ident "{" Expr "}"
Children  := "{" Node* "}"
TextNode  := "text" String | String
ExprNode  := "@" Expr
Directive := "#" Ident (("(" Args ")") | Args)? Children?
Value     := String | Number | true | false | null
Expr      := Raw JS until delimiter
